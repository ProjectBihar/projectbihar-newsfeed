# Plan: Fix Geo-Fencing Leaks in Bihar Scraper

## Context

The current geo-fencing logic has three real bugs that cause articles to be incorrectly dropped or accepted. The proposed patch addresses all three in a minimal, well-scoped set of changes across three files.

## Leak Analysis — Current vs Proposed

### Leak 1: Multi-word Hindi terms never match (`token-match.ts`)

**Current (line 34-37):** Splits Hindi text on non-Devanagari chars into individual tokens, then checks `tokens.some(tok => tok === t)`. For multi-word terms like `"बिहार शरीफ"`, the split produces `["बिहार", "शरीफ"]` — no single token equals the full term, so it **never matches**.

**Proposed:** Replace with regex bounded-substring matching: `/(^|[^ऀ-ॿ])<term>([^ऀ-ॿ]|$)/i`. This correctly matches contiguous multi-word Hindi terms while still blocking partial matches like `"कर"` inside `"विशेषकर"`.

**Impact:** Currently, multi-word Hindi terms in `BIHAR_GEO_DICTIONARY` (e.g., `"बिहार शरीफ"`, `"पश्चिम चंपारण"`, `"मध्य प्रदेश"`, `"उत्तर प्रदेश"`) silently fail to match. This affects both Bihar detection AND non-Bihar state detection (Hindi entries in `NON_BIHAR_STATES` like `"उत्तर प्रदेश"` also silently fail).

### Leak 2: No dateline signal (`article-extractor.ts`)

**Current:** `ArticleData` has `headline`, `synopsis`, `published_timestamp`. No structural location signal.

**Proposed:** Add `dateline: string | null` field. Extract from first `<p>` using two regex patterns:
- English: `PATNA:` or `PATNA (PTI):`
- Hindi: `पटना,` or `पटना।`

**Impact:** Dateline is a strong structural signal independent of keyword scanning. An article datelined from Patna is almost certainly Bihar-relevant, regardless of what keywords appear in the body.

### Leak 3: Dedicated sources run through the geo-fence (`index.ts`)

**Current (line 192-216):** `isBiharRelevant(headline, synopsis)` — no dateline, no source-type awareness. Every article from every source runs through the same keyword scan.

**The problem:** Sources like Patna Press are tagged `type: 'bihar-dedicated'` in `sources.ts` with a comment saying they should bypass the geo-fence. But `isBiharRelevant` never checks this. A Patna Press article about a Bihar-UP river dispute mentioning `"Uttar Pradesh"` gets hard-vetoed by the non-Bihar state check.

**Proposed:** 
1. Replace `isBiharRelevant` with a two-function approach:
   - `biharRelevanceScore(headline, synopsis, dateline)` → net score (`biharHits - nonBiharHits`, with dateline bonus ±5)
   - `isBiharRelevant(headline, synopsis, dateline, isDedicatedSource)` → bypasses scan for dedicated sources, otherwise `score > 0`
2. Call site becomes: `isBiharRelevant(data.headline, data.synopsis, data.dateline, source.type === 'bihar-dedicated')`

**The net-score approach also fixes:** Articles about Bihar migrants in UP, or Bihar-UP river disputes, that mention non-Bihar entities but ARE Bihar-relevant — these now survive if Bihar signals outweigh non-Bihar signals.

### Cleanup: Removed arrays

The proposed patch removes `BIHAR_STATE_NAMES`, `BIHAR_DISTRICTS`, and `NON_BIHAR_FALSE_POSITIVE_CITIES` from `index.ts`. These were subsets/overlaps of `BIHAR_GEO_DICTIONARY` and `NON_BIHAR_STATES`. The score logic uses only the two main dictionaries, which is cleaner.

## Files to Modify (3 files)

### 1. `src/scraper/token-match.ts` — Fix Hindi multi-word matching

Replace the Hindi branch of `matchesToken` (lines 33-37):

```ts
// BEFORE (broken for multi-word):
if (/[\u0900-\u097F]/.test(t)) {
    const tokens = lower.split(/[^ऀ-ॿ]+/).filter(Boolean);
    return tokens.some((tok) => tok === t);

// AFTER (correct for multi-word):
if (/[\u0900-\u097F]/.test(t)) {
    const pattern = new RegExp(`(^|[^ऀ-ॿ])${escapeRegex(t)}([^ऀ-ॿ]|$)`, 'i');
    return pattern.test(lower);
}
```

### 3. `src/scraper/index.ts` — Net-score relevance + dedicated source bypass

**a)** Remove `BIHAR_STATE_NAMES` (lines 140-142), `BIHAR_DISTRICTS` (lines 145-163), and `NON_BIHAR_FALSE_POSITIVE_CITIES` (lines 165-190).

**b)** Replace `isBiharRelevant` function (lines 192-216) with:
```ts
function biharRelevanceScore(headline: string, synopsis: string, dateline: string | null): number {
  const text = `${headline} ${synopsis}`;
  const biharHits = BIHAR_GEO_DICTIONARY.filter((t) => matchesToken(text, t)).length;
  const nonBiharHits = NON_BIHAR_STATES.filter((t) => matchesToken(text, t)).length;
  let score = biharHits - nonBiharHits;

  if (dateline) {
    if (matchesAnyToken(dateline, BIHAR_GEO_DICTIONARY)) score += 5;
    else if (matchesAnyToken(dateline, NON_BIHAR_STATES)) score -= 5;
  }

  return score;
}

function isBiharRelevant(
  headline: string,
  synopsis: string,
  dateline: string | null,
  isDedicatedSource: boolean
): boolean {
  if (isDedicatedSource) return true;
  return biharRelevanceScore(headline, synopsis, dateline) > 0;
}
```

**c)** Update call site (line 283):
```ts
// BEFORE:

// AFTER:
if (!isBiharRelevant(data.headline, data.synopsis, data.dateline, source.type === 'bihar-dedicated')) {
```

## Blast Radius

- **`token-match.ts`:** API unchanged (`matchesToken`, `matchesAnyToken`, `escapeRegex` signatures stay the same). Consumers in `noise-filter.ts`, `classifier.ts`, `pipeline.ts`, `block-filter.ts` are unaffected.
- **`article-extractor.ts`:** `ArticleData` gains one optional field. Only consumer is `index.ts`, which gets updated.
- **`index.ts`:** `isBiharRelevant` gains two new parameters. Only caller is the `scrapeSource` function in the same file.
